### 2.2 障碍环境建模

状态：`已完成`

已确认：

- 障碍信息是对真实海域物理状况的抽象
- 中间风险区必须保证至少存在 `2` 条不规则可通过航道
- 中间风险区不预定义固定航道
- 障碍数量不是核心目标，核心目标是形成合理的通行约束与路径规划难度
- 远海区设置少量孤立障碍或风险点
- 远海区“孤立障碍”表示局部实体通行障碍，主要回答“能不能过”
- 远海区“风险点”表示局部非实体高风险海域，主要回答“值不值得过、是否应尽量避开”
- 近海区设置 `2` 个低优先级基础监测点
- 远海区设置 `5` 个任务热点
- 基础监测点与任务热点在对应分区内随机生成，并避开障碍物与风险区
- 每次仿真环境采用伪随机生成
- 对 `USV`：障碍作为不可穿越区或高代价区域
- 对 `UAV`：障碍保留区域语义，不直接限制飞越

当前代码结果：

### 2.3 离散栅格网络与覆盖映射

状态：`已完成`

已确认：

- 覆盖结果需要映射到离散网络
- 第一版离散网络采用规则矩形栅格
- 栅格尺寸不直接等于智能体本体大小
- 栅格尺寸主要由传感器覆盖分辨率、任务精度和计算复杂度决定
- 智能体本体尺寸与栅格尺寸分离建模
- 第一版栅格边长采用 `25 m`
- 第一版离散栅格网络规模为 `40 x 40`
- 第一版栅格显式保存区域属性、障碍属性、风险区属性、基础监测点属性与任务热点属性
- 网格覆盖判定采用“中心点落入 footprint 即覆盖”
- 不计算 footprint 与整个网格的精确相交面积
- 第一版覆盖映射在现有栅格网络上扩展，不单独建立第二套离散结构

### 2.6 智能体数学建模

状态：`已完成（第一阶段存档）`

已确认：

- 在覆盖映射与后续动态仿真之前，需要先补充智能体的最小可用数学建模
- 该数学建模只需支撑位置更新、朝向表达、探测半径、覆盖半径和基础任务状态
- 第一版优先采用简化运动学模型，不在本阶段引入高保真控制级动力学
- 后续若考虑控制器因素，整体应按“任务层 -> 规划层 -> 控制器层 -> 动力学层”的分层闭环结构推进
- `UAV` 控制与动力学建模可先简化，重点体现快速机动和稳定跟踪能力
- `USV` 控制与动力学建模应重点体现转向受限、跟踪误差和绕障难度
- 后续研究中应保留“参考轨迹”和“实际轨迹”的区分，用于分析控制器和动力学对覆盖表现的影响
- 当前覆盖方法研究只基于第一阶段动力学与控制闭环展开
- 第二阶段与第三阶段暂不进入当前实现范围，待第一阶段覆盖研究推进顺利后再评估是否启动

- 当前数学模型第一阶段已完成并作为当前项目基线存档，已能支撑覆盖映射与后续异构多智能体覆盖方法研究

阶段规划：

- 第一阶段：

- 第二阶段：
  - 进一步增强 `USV` 动力学真实性
  - 引入更明显的转向限制、跟踪误差和轨迹偏差

  - 当前暂缓
- 第三阶段：
  - 研究不同控制器与动力学差异对协同覆盖效果的影响
  - 分析跟踪误差、转向约束和执行偏差对任务效率与覆盖质量的作用
  - 当前暂缓

### 2.7 信息地图

状态：`已完成（第一版）`

已确认：

- 信息地图直接复用现有 `40 x 40` 栅格，以单个栅格作为最小信息单元
- 当前阶段不额外建立区域级信息图
- 信息失效随时间流逝发生
- 具体失效时长暂不固定，作为后续可调参数
- 超过失效阈值后，该栅格状态记为“过时已知”
- 栅格重新被观测后，信息立即刷新为有效
- 热点真实状态与系统认知状态分离建模
- 真实状态用于表示环境里是否真实存在热点
- 认知状态用于表示系统当前知道到什么程度
- 建议的热点认知状态为：
  - `none`
  - `suspected`
  - `confirmed`
  - `false_alarm`
- 当前阶段热点生成规则为：
  - 近海区和远海区都允许刷新热点
  - 近海区热点刷新概率更小
  - 热点允许持续存在
  - 多热点并存上限当前尚未最终确定，后续以参数形式补充

建议的栅格动态信息字段：

- 观测时效字段

  - `information_age`
  - `is_information_valid`
  - `is_stale_known`
- 环境认知字段

  - `known_obstacle`
  - `last_updated_by`
- 基础任务字段
  - `has_baseline_task`
  - `baseline_task_priority`
  - `baseline_last_served_step`
- 热点真实状态字段
  - `ground_truth_hotspot`
  - `ground_truth_hotspot_id`
  - `ground_truth_hotspot_created_step`
- 热点认知状态字段
  - `known_hotspot_state`
  - `known_hotspot_id`
  - `suspected_by`
  - `confirmed_by`
  - `suspicion_step`
  - `confirmation_progress`
- 执行状态字段
  - `assigned_agent_id`
  - `task_status`

当前阶段最核心的第一版字段：

- `last_observed_step`
- `information_age`
- `is_information_valid`
- `has_baseline_task`
- `ground_truth_hotspot`
- `known_hotspot_state`
- `confirmation_progress`
- `assigned_agent_id`

热点状态转移规则：

- 情况一：真实热点
  - 环境中某栅格生成真实热点，但系统未必立即知道
  - `UAV` 优先对其进行初步探测，并将其标记为 `suspected`
  - `USV` 被派往目标栅格附近近距离停留观测若干步
  - 若确认条件满足，则状态从 `suspected` 转为 `confirmed`
- 情况二：假热点
  - `UAV` 可先将其标记为 `suspected`
  - `USV` 近距离复查后，若不满足真实热点条件，则状态转为 `false_alarm`

- 热点消失/清除机制
- 更复杂的观测与误报模型

- 同一类算法按包分层，不同算法默认采用“一个算法一个 `.py` 文件”的组织方式
- 文件名应尽量直接体现算法名称或策略名称，便于后续实验切换与对比

### 2.10 智能体动态仿真

状态：`待实现`

已确认：

- 总流程采用：
  `环境与感知 -> 任务层 -> 路径层 -> 执行层 -> 覆盖反馈 -> 再回到任务层`
- 智能体在连续二维海域中运动、规划与协同
- `USV/UAV` 先对目标区域进行覆盖搜索
- 区域被搜索后可获得对应环境与任务信息，且当前阶段默认所有智能体实时共享区域信息
- 当前阶段不考虑通信约束，默认不引入通信半径、延迟、丢包和带宽限制
- 后续任务热点应主要由 `UAV` 先行检测
- `USV` 在自身探测过程中也可以独立发现任务热点
- `UAV` 发现任务热点后，再引导 `USV` 前往复查或精细执行
- 基础监测区属于长期已知、周期维护的任务
- 任务热点属于动态刷新任务
- 在仿真中通过不同探测半径、不同发现能力和不同确认能力体现 `UAV/USV` 的异构热点发现机制
- `UAV` 更适合将热点从“未发现”推进到“疑似热点”
- `USV` 更适合将热点从“疑似热点”推进到“已确认热点”
- 热点状态后续按 `未发现 -> 疑似热点 -> 已确认热点` 的流程推进
- 近海区不作为热点主区域，但保留 `2` 个低优先级基础监测点
- 远海区作为主要任务热点区，设置 `5` 个任务热点
- 任务热点支持随机刷新
- 若一个区域长时间未被监测，则该区域信息会失效；若此时区域内刷新新任务，系统不会立即知晓
- 若区域持续处于监测状态，则任务刷新后系统可以及时发现并响应
- 后续需要考虑 `USV` 受损事件，系统应在受损后动态调整，确保剩余智能体继续执行任务
