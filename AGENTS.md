# AGENTS.md

本文件用于指导 Codex 在本仓库中的后续 Python 编程、重构、测试与文档编写工作。

## 0. 项目定位

本仓库默认按 Python 项目处理。

除非用户后续明确指定，否则 Codex 应遵守以下默认判断：

- 主语言：Python 3.11+
- 包管理优先级：`pyproject.toml` > `requirements.txt`
- 测试框架优先：`pytest`
- 代码目录优先：`src/`
- 测试目录优先：`tests/`
- 虚拟环境目录优先：`.venv/`

## 1. 目标

Codex 在本项目中的核心职责：

- 基于用户需求直接落地代码，而不是只给方案。
- 优先产出可运行、可验证、可维护的实现。
- 在信息不足时，先检查仓库现状，再做最小合理假设。
- 保持修改聚焦，避免无关重构。

如果用户没有明确要求“只分析不修改”，默认进入可执行交付模式。

## 2. 工作方式

### 2.1 开始任务前

每次开始前先做以下事情：

1. 先完整阅读本文件 `AGENTS.md`。
2. 如果当前工作目录或目标子目录下存在更具体的 `AGENTS.md`，继续优先阅读并遵守更细粒度规则。
3. 阅读当前目录结构与关键配置文件。
4. 判断 Python 项目的入口文件、依赖管理方式、测试方式、运行方式。
5. 如仓库信息不足，采用最小可行方案，不凭空臆造复杂架构。

### 2.2 执行原则

- 优先先做、再解释，除非用户明确要求先讨论。
- 优先小步快跑，先完成主路径，再补充边界。
- 优先兼容现有代码风格，不强推个人偏好。
- 优先修改真正相关的文件，避免顺手大改。
- 发现潜在风险时要明确指出，但不要空泛阻塞。

### 2.3 交付要求

完成任务时尽量同时给出：

- 实际代码修改
- 必要的测试或验证
- 简洁说明
- 未完成项或风险说明

## 3. 代码实现约束

### 3.1 通用要求

- 默认使用清晰、直接、易维护的实现。
- 避免过度封装、过早抽象、无意义设计模式。
- 变量、函数、类名必须表达真实语义。
- 优先写人能读懂的代码，而不是炫技代码。
- 非必要不引入新依赖。
- 非必要不新增全局状态。

### 3.2 Python 代码要求

- 优先兼容 Python 3.11 及以上语法。
- 默认遵循 PEP 8，并保持风格一致。
- 优先使用类型注解，至少为公共函数、核心类、关键返回值补充类型。
- 模块职责保持单一，避免单文件堆积过多不相关逻辑。
- 优先使用标准库，只有在明显提高开发效率或可维护性时才引入第三方库。
- 异常处理要具体，避免无差别 `except Exception`，除非是在边界层统一兜底。
- 涉及 I/O、网络、配置、数据库时，应将核心逻辑与副作用分离，便于测试。

### 3.3 编码规范

后续实际编写代码时，必须优先满足以下要求：

- 可读性强
- 逻辑清晰
- 实现简洁
- 避免冗余

具体执行规则如下：

- 一个模块只处理一类职责，避免环境、智能体、规划、可视化逻辑混杂。
- 一个函数只做一件事，输入、输出和副作用要明确。
- 命名必须表达真实语义，避免含糊缩写和难以理解的临时命名。
- 优先使用直白实现，不为了“高级感”引入不必要的抽象、设计模式或复杂控制流。
- 能用简单函数解决的问题，不强行设计大而全的类结构。
- 避免重复代码；相同逻辑应适度提取复用，但不要过早抽象。
- 避免超长函数、超大文件和过深嵌套，优先采用小步拆分与早返回。
- 参数、阈值、尺寸、速度、半径等信息优先集中管理，避免魔法数字散落在代码中。
- 数据结构优先清晰稳定，配置、状态、结果优先使用 `dataclass` 或等价清晰结构表达。
- 核心算法逻辑、环境建模逻辑、仿真调度逻辑、可视化逻辑应尽量解耦。
- 注释只在必要时添加，重点解释“为什么这样做”，而不是重复代码表面含义。
- 不保留临时调试输出、废弃分支、无意义兼容代码或未使用封装。
- 若实现存在多个可行方案，优先选择最容易被后续维护者读懂和修改的版本。

### 3.4 修改策略

- 优先复用已有模块、函数、配置和工具链。
- 如果已有代码质量一般，先局部修复，不立即推倒重来。
- 如果需要重构，应先保证行为不回归。
- 对用户未要求的功能不主动扩展。

### 3.5 注释要求

- 只在“代码意图不明显”时添加简短注释。
- 注释解释“为什么这么做”，不要重复“代码做了什么”。
- 不写噪音注释。

### 3.6 函数与类设计

- 函数优先短小、单一职责、输入输出清晰。
- 不要无必要地写超大类；能用简单函数解决时，不强行面向对象。
- 数据结构优先考虑 `dataclass`、`TypedDict`、`Enum` 等 Python 原生或标准方案。
- 公共接口应保持稳定，变更时同步更新调用方和文档。

## 4. 文件与目录约定

当项目还未成型、需要从零开始创建结构时，优先遵守以下 Python 项目结构：

- `src/`：主业务代码
- `tests/`：测试代码
- `docs/`：补充文档
- `scripts/`：辅助脚本
- `configs/` 或根目录配置文件：环境与工程配置
- `pyproject.toml`：项目配置、依赖、工具配置优先入口
- `.venv/`：本地虚拟环境
- `.env.example`：环境变量示例

如果项目后续采用特定框架，应优先遵循框架惯例，而不是强行套用上述目录。

### 4.1 推荐初始化方向

从零开始时，优先形成如下最小结构：

```text
.
|-- AGENTS.md
|-- pyproject.toml
|-- README.md
|-- .env.example
|-- src/
|   `-- <package_name>/
|       `-- __init__.py
`-- tests/
    `-- test_smoke.py
```

如果用户后续指定 FastAPI、Flask、CLI、爬虫、数据处理、AI 工程等方向，再按实际场景扩展。

## 5. 测试与验证

Codex 在实现后应尽量做验证，而不只是假设代码正确。

### 5.1 最低要求

- 新增功能：至少验证主流程。
- 修复 bug：至少覆盖复现路径和修复路径。
- 重构：至少确认原有行为未破坏。

### 5.2 测试策略

- 优先运行与本次修改直接相关的测试。
- 如果没有测试框架，至少执行最小可行的本地验证。
- 如果因环境限制无法运行测试，必须明确说明未验证原因。

### 5.3 Python 测试约定

- 默认使用 `pytest`。
- 测试文件命名优先：`test_*.py`。
- 单元测试优先覆盖纯函数、业务规则、边界条件和错误路径。
- 涉及文件、网络、时间、随机数、外部服务时，优先使用 mock、fixture 或依赖注入隔离副作用。
- 修复 bug 时，优先先补一个能复现问题的测试，再修复实现。

## 6. 文档策略

以下情况需要同步更新文档：

- 新增公共能力、接口、命令或配置项
- 修改使用方式
- 引入关键限制、约束或兼容性变化

文档应简明、准确、可直接给下一位开发者使用。

### 6.1 Python 项目文档最低要求

如果仓库进入可运行阶段，README 至少应包含：

- 项目简介
- Python 版本要求
- 安装方式
- 运行方式
- 测试方式
- 关键环境变量说明

### 6.2 README 同步要求

- 每次完成实际代码新增、删除或结构性调整后，必须同步更新 `README.md` 中的项目工程目录说明。
- `README.md` 中展示的目录结构应与仓库当前真实结构保持一致。
- 如果本次改动没有影响工程结构，可不重复改写其他 README 内容，但应主动检查目录说明是否仍然准确。

## 7. 与用户协作方式

- 默认使用中文沟通，除非用户要求英文。
- 先理解“用户要结果还是要方案”。
- 若问题存在多条明显路线，只在确有权衡时再询问用户。
- 不把本该自己完成的探索工作转交给用户。
- 如果需要做假设，应在完成后明确写出假设内容。
- 如果是 Python 工程初始化任务，优先直接搭出可运行骨架。

## 8. 输出格式要求

完成任务后的回复尽量包含：

1. 做了什么
2. 改了哪些关键文件
3. 如何验证
4. 还有什么风险或后续建议

回复要简洁，避免冗长流水账。

## 9. 决策优先级

当多个方案都可行时，按以下优先级决策：

1. 正确性
2. 可维护性
3. 与现有项目一致性
4. 实现成本
5. 性能优化

除非性能是明确目标，否则不要为了微小性能收益牺牲可读性。

## 10. 从零启动项目时的默认要求

如果用户后续让 Codex 从空仓库启动项目，默认应补齐这些基础能力：

- 清晰的目录结构
- Python 启动方式
- 环境变量示例
- 基本 README
- 基本 `pytest` 测试能力
- 基础 lint 或格式化能力
- 基本依赖声明

但仍然坚持最小可行原则，不一次性堆砌完整“工业模板”。

### 10.1 默认工具链

如果用户没有指定工具，优先采用以下组合：

- 依赖配置：`pyproject.toml`
- 测试：`pytest`
- 格式化：`ruff format` 或 `black`
- lint：`ruff`

若仓库已存在其他工具链，应优先与现有工具保持一致。

## 11. 禁止事项

- 不凭空伪造未验证的接口、数据结构或第三方能力。
- 不在没有依据时声称“已经测试通过”。
- 不随意删除用户已有代码。
- 不进行与当前任务无关的大规模格式化。
- 不将临时调试代码遗留在最终提交中。
- 不在没有明确收益时引入复杂异步、元类、装饰器链或过度动态化写法。

## 12. 维护方式

当项目逐步成型后，应持续补充本文件，重点更新：

- 技术栈
- 目录结构
- 启动命令
- 测试命令
- 编码约定
- 业务边界
- 常见陷阱
- Python 包名、入口模块、环境管理方式

目标是让后续进入本仓库的 Codex 能快速理解项目并稳定产出。
